## 2. 데이터/인프라

### 2.1 기존 스키마 (변경 없음)
M0 `images` 테이블:
```sql
create table public.images (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references public.users(id) on delete cascade,
  page_id uuid references public.pages(id) on delete set null,
  storage_path text not null,
  encrypted boolean not null default false,
  size_bytes integer,
  created_at timestamptz not null default now()
);
```
RLS는 `images_(select|insert|update|delete)_own` 4개 모두 `auth.uid() = user_id` (M0).

### 2.2 새 마이그레이션 — Storage bucket
파일: `supabase/migrations/20260514020000_storage_images_bucket.sql`

```sql
-- bucket 생성 (public read, RLS-driven write)
insert into storage.buckets (id, name, public)
values ('images', 'images', true)
on conflict (id) do nothing;

-- 사용자 자신의 폴더에만 INSERT/UPDATE/DELETE 허용
-- path 첫 segment가 user_id여야 함: e.g. "{auth.uid()}/{uuid}.webp"
create policy "images_storage_insert_own"
  on storage.objects for insert
  with check (
    bucket_id = 'images'
    and auth.uid()::text = (storage.foldername(name))[1]
  );

create policy "images_storage_update_own"
  on storage.objects for update
  using (
    bucket_id = 'images'
    and auth.uid()::text = (storage.foldername(name))[1]
  );

create policy "images_storage_delete_own"
  on storage.objects for delete
  using (
    bucket_id = 'images'
    and auth.uid()::text = (storage.foldername(name))[1]
  );

-- SELECT는 public bucket이라 정책 없이도 누구나 읽을 수 있음 (URL 추측 어려움)
```

### 2.3 한도
- 상수: `STORAGE_LIMIT_BYTES = 100 * 1024 * 1024` (100MB)
- 정의 위치: `lib/images/constants.ts`

### 2.4 비공개 페이지 이미지 (defer 명시)
M3.2는 `encrypted=false`만 INSERT. 비공개 페이지에 추가된 이미지도 일단 public URL로 노출됨. **M5에서 'images-private' bucket + DEK 암호화로 대체.** 본 spec 범위 외이지만 invariant 명시.

## 3. 클라이언트 흐름

### 3.1 압축 유틸
파일: `lib/images/compress.ts`

```ts
export interface CompressResult {
  blob: Blob;        // image/webp
  sizeBytes: number; // blob.size
  filename: string;  // "{uuid}.webp"
}

export async function compressImage(file: File): Promise<CompressResult>
```

동작:
- `browser-image-compression` 의존 (`maxWidthOrHeight: 1920, quality: 0.8, fileType: 'image/webp', useWebWorker: true`)
- 압축 결과가 원본보다 크면 원본 사용 (단 형식은 WebP로 통일하기 위해 그래도 변환된 WebP 사용 — 일관성)
- HEIC/SVG 등 압축 불가 형식은 catch → throw `Error('이미지 형식이 지원되지 않습니다.')`
- filename은 `crypto.randomUUID() + '.webp'`

### 3.2 BlockNote uploadFile 핸들러
파일: `lib/images/upload-handler.ts`

```ts
import { createSupabaseBrowserClient } from '@/lib/supabase/client';
// (note: 함수명은 lib/supabase/client.ts 의 createSupabaseBrowserClient)

export function createUploadFileHandler(pageId: string) {
  return async (file: File): Promise<string> => {
    // 1. preflight quota
    const usage = await getStorageUsage();
    if (!usage.ok) throw new Error(usage.error);
    if (usage.data.usedBytes + file.size > usage.data.limitBytes) {
      // 모달 표시는 외부 이벤트로 (window.dispatchEvent 'storage-quota-exceeded')
      window.dispatchEvent(new CustomEvent('storage-quota-exceeded'));
      throw new Error('저장 용량 한도를 초과했어요.');
    }

    // 2. compress
    const { blob, sizeBytes, filename } = await compressImage(file);

    // 3. storage upload
    const supabase = createSupabaseBrowserClient();
    const { data: { user } } = await supabase.auth.getUser();
    if (!user) throw new Error('로그인이 필요합니다.');

    const storagePath = `${user.id}/${filename}`;
    const { error: uploadErr } = await supabase.storage
      .from('images')
      .upload(storagePath, blob, { contentType: 'image/webp', upsert: false });
    if (uploadErr) throw new Error(uploadErr.message);

    // 4. metadata + post-check
    const result = await recordImage({ storagePath, sizeBytes, pageId });
    if (!result.ok) {
      // post-check 실패 시 Storage에서 즉시 삭제 (race lost)
      await supabase.storage.from('images').remove([storagePath]).catch(() => {});
      if (result.error === 'quota') {
        window.dispatchEvent(new CustomEvent('storage-quota-exceeded'));
      }
      throw new Error(result.error === 'quota' ? '저장 용량 한도를 초과했어요.' : result.error);
    }

    return result.data.url;
  };
}
```

`window.dispatchEvent` + `addEventListener` 패턴으로 모달 트리거 — `<QuotaExceededModalMount />`가 layout에서 listen.

### 3.3 BlockNote wiring
파일: `app/(app)/p/[pageId]/PageBody.tsx` (기존 수정)

## 4. 서버 액션 (`lib/actions/images.ts`)

`'use server'` 파일. 모든 export는 async function. Schema는 export 안 함 (M2.4 패턴 — 필요 시 `images-schemas.ts`로 분리).

### 4.1 `getStorageUsage()`
```ts
export async function getStorageUsage(): Promise<ActionResult<{
  usedBytes: number;
  limitBytes: number;
}>>
```
- `select('size_bytes').eq('user_id', me.id)` → sum
- limitBytes = `STORAGE_LIMIT_BYTES`

### 4.2 `recordImage(input)`
```ts
export async function recordImage(input: {
  storagePath: string;
  sizeBytes: number;
  pageId: string | null;
}): Promise<ActionResult<{ id: string; url: string }> | { ok: false; error: 'quota' | string }>
```
- Zod: storagePath `^[a-f0-9-]+/[a-f0-9-]+\.webp$`, sizeBytes positive int, pageId nullable uuid v4
- post-check: `select sum(size_bytes) from images where user_id = auth.uid()` + new size → if > limit, return `{ok:false, error:'quota'}` (호출자가 Storage 정리)
- INSERT row, public URL 계산 (`supabase.storage.from('images').getPublicUrl(storagePath).data.publicUrl`)
- revalidatePath(`/p/${pageId}`) if pageId
- return `{ok: true, data: {id, url}}`

### 4.3 `deleteImages(ids)`
```ts
export async function deleteImages(ids: string[]): Promise<ActionResult<{ deletedCount: number }>>
```
- Zod: `z.array(z.string().uuid()).min(1)`
- `select('storage_path').in('id', ids)` → paths 수집
- Storage `.remove(paths)` 호출 (실패해도 진행)
- DB `delete().in('id', ids)` (RLS가 본인 것만)
- revalidatePath('/settings/storage')
- return `{ok:true, data:{deletedCount}}`

### 4.4 `listOrphanImages()`
```ts
export async function listOrphanImages(): Promise<ActionResult<{ ids: string[] }>>
```
- 내 모든 페이지(`pages where deleted_at is null and user_id = auth.uid()`)의 `content` jsonb 배열을 fetch
- 각 페이지 content를 재귀 탐색해 `block.type === 'image'` 블록의 `props.url` (BlockNote default image block) 수집
- URL → storage_path 변환 (URL에서 `/storage/v1/object/public/images/{path}` 추출)
- 내 모든 image row와 비교 → 어디에도 없는 image id 수집
- 휴지통(deleted_at != null) 페이지의 이미지는 orphan으로 간주 (참조 안 살아있음)

## 7. 에러/엣지 케이스

| 상황 | 처리 |
|---|---|
| 압축 실패 (HEIC, SVG, 손상) | `throw Error('이미지 형식이 지원되지 않습니다.')` → BlockNote가 placeholder 유지 |
| 네트워크 오류 (Storage upload) | throw → BlockNote 재시도 안내. 부분 업로드 잔여 없음 (Storage atomic) |
| 쿼터 race (여러 탭 동시 업로드) | recordImage post-check가 catch → Storage `.remove()` + 모달 |
| 빈 파일 / 0바이트 | compress 단계에서 throw |
| pageId 없는 컨텍스트 (예: 대시보드) | 본 phase는 page 안에서만 업로드 → pageId 항상 있음. 대시보드 이미지 업로드 unsupported. |
| public URL이 깨짐 | BlockNote default 이미지 블록이 fallback 처리 |
| 동시 같은 파일명 (UUID 충돌) | crypto.randomUUID() — 사실상 0 |

## 11. 검증 시나리오 (수동)

1. 새 페이지 → 슬래시 메뉴 `/이미지` → 5MB 사진 추가 → ~1MB WebP로 자동 압축 후 표시
2. /settings/storage → 사용량 47.3 MB / 100 MB 표시
3. 한도 직전(95MB)에 10MB 이미지 추가 시도 → M-009 모달 자동 표시 + 정리 버튼
4. M-009에서 5장 선택 삭제 → 다시 시도 가능
5. 페이지 본문에서 이미지 블록 삭제 → S-018에서 "사용 안 하는" 탭에 노출
6. 페이지 영구삭제 (휴지통 비우기) → 그 페이지의 이미지가 orphan으로 분류
7. 다른 사용자 폴더 path로 직접 upload 시도 → Storage RLS reject
