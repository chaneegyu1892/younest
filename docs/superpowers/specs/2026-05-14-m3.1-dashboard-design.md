## 3. 데이터 가져오기

### 3.1 추가할 쿼리: `fetchRecentPages`
파일: `lib/supabase/queries/pages.ts`

시그니처:
```ts
export async function fetchRecentPages(limit: number): Promise<PageNode[]>
```

동작:
- `from('pages').select(PAGE_SELECT)` (기존 SELECT 재사용)
- `.is('deleted_at', null)`
- `.order('updated_at', { ascending: false })`
- `.limit(limit)`
- 반환은 fetchUserPages와 동일하게 `PageNode[]`로 캐스팅 (postgrest-js 추론 한계 → `as unknown as PageNode[]`)

`PAGE_SELECT`는 이미 `updated_at` 포함 (M2.4까지 추가된 컬럼들). 새 컬럼 불필요.

### 3.2 즐겨찾기는 별도 쿼리 없음
`page.tsx`에서 `fetchUserPages()` 결과를 그대로 받아 client-side filter:
```ts
const favorites = pages.filter(p => p.is_favorite).slice(0, 8);
```
사이드바와 동일한 데이터셋을 한 번 더 받는 비용은 RSC라 캐시되어 무시 가능.

### 3.3 `fetchRecentPages`는 별도 호출
`fetchUserPages`는 트리 구성용으로 모든 페이지를 가져오지만 정렬이 다르고 잠재적으로 큰 데이터셋. 최근 10건은 별도 쿼리가 명확.

## 4. 빠른 작성 (QuickCreate)

### 4.1 UI
3 카드 그리드 (`grid grid-cols-3 gap-3`):
| 버튼 | title | icon |
|---|---|---|
| 메모 | `"메모"` | `📝` |
| 일기 | `"YYYY-MM-DD 일기"` (`new Date().toISOString().slice(0, 10)`) | `📖` |
| 기도제목 | `"기도제목"` | `🙏` |

각 카드는 큰 아이콘 + 라벨 + hover bg. `disabled`는 transition 중에만.

### 4.2 동작
client component, `useTransition` + `useRouter`.

```ts
const [pending, start] = useTransition();
const router = useRouter();

function handleCreate(template: 'memo' | 'diary' | 'prayer') {
  start(async () => {
    const { title, icon } = getTemplate(template);
    const result = await createPage({ parentPageId: null, type: 'document', title, icon });
    if (!result.ok) {
      toast.error(`페이지 생성 실패: ${result.error}`);
      return;
    }
    // 성공: router.push 후 사이드바·대시보드는 createPage가 호출한 revalidatePath로 자동 동기화
    router.push(`/p/${result.data.id}`);
  });
}
```

`getTemplate`는 컴포넌트 파일 내 작은 함수. 일기 title은 호출 시점에 today를 ISO 자르기.

### 4.3 createPage 스키마 확장
현재 `createSchema`는 `{parentPageId, type, title}`. **`icon: z.string().max(16).nullable().optional()` 추가**.
INSERT에 `icon: parsed.data.icon ?? null` 포함.
기존 호출부(`PageTreeNode` 등)는 icon 미지정 → optional이라 호환 유지.

## 5. 즐겨찾기 위젯 (FavoritesWidget)

RSC. `pages: PageNode[]` prop.

### 5.1 렌더링
```
favorites = pages.filter(is_favorite).slice(0, 8)
```
- `favorites.length === 0` → return null (제목까지 숨김)
- 그 외:
  - 섹션 제목: `<h2>즐겨찾기</h2>` + 카운트 배지 `(N)` (count는 slice 전 favorites.length, 단 8 이상이면 `8+`로 표기 안 함 — slice된 N 그대로)
  - `grid grid-cols-2 sm:grid-cols-3 md:grid-cols-4 gap-3`
  - 각 카드: `<Link href={`/p/${id}`}>` + icon (없으면 `⭐`) + title (없으면 `"제목 없음"`, truncate)

## 6. 최근 수정 위젯 (RecentWidget)

RSC. `pages: PageNode[]` prop (이미 정렬·limit 적용된 것).

### 6.1 렌더링
- `pages.length === 0` → return null
- 그 외:
  - 섹션 제목: `<h2>최근 수정</h2>` + 카운트 배지
  - 단순 `<ul>` 리스트, 각 row: icon (없으면 `📄`) + title (truncate) + `<time>` 상대시간 (오른쪽 정렬, `text-text-tertiary text-caption`)
  - row hover bg, Link to `/p/[id]`

### 6.2 상대시간 유틸
파일: `lib/utils/relative-time.ts`

```ts
export function formatRelativeTime(iso: string, now: Date = new Date()): string
```

규칙 (한국어, `Intl.RelativeTimeFormat('ko', { numeric: 'auto' })`):
- < 60초: `"방금 전"`
- < 60분: `N분 전`
- < 24시간: `N시간 전`
- < 7일: `N일 전`
- < 4주: `N주 전`
- 그 외: `Intl.DateTimeFormat('ko', { dateStyle: 'medium' })` (`2026. 5. 14.`)

테스트는 `now` 인자로 결정성 확보.

## 11. 검증 시나리오

1. **빈 계정 (페이지 0)**: 인사 + 3 버튼만. 버튼 클릭 → 새 페이지 즉시 이동 + 사이드바·대시보드 동기화.
2. **즐겨찾기 0, 일반 5**: 즐겨찾기 섹션 안 보이고, 최근 수정 5건 표시.
3. **즐겨찾기 12건**: 8개만 카드로 표시, 카운트 배지는 8.
4. **최근 수정 정렬 검증**: 페이지 A 편집 → 대시보드 새로고침 시 A가 최상단.
5. **빠른 작성 일기 title**: 오늘 날짜로 생성 (다음날 새로 만들면 다른 날짜).
6. **createPage icon 적용**: DB row의 icon 컬럼이 이모지로 채워짐. 사이드바 트리에도 즉시 노출.
7. **상대시간 경계**: 1분 직전/직후, 24시간 경계, 7일 경계, 4주 경계 (수동 검증은 1-2개만; 단위 테스트로 커버).

## 13. 빠른 작성 v2 — 프로젝트 + 자식 누적 구조 (2026-05-14 추가)

> 처음 v1은 매 클릭마다 flat 새 루트 페이지를 생성. 사이드바가 폭주하는 문제 발견 → 부모 프로젝트 페이지 + 자식 누적으로 변경.

### 13.1 동작
- **메모/일기/기도제목 모두 동일 패턴**: 각 템플릿마다 부모 프로젝트 페이지 1개를 두고, 클릭 시 그 안에 자식 페이지를 생성.
- 부모가 없으면 자동 생성, 있으면 재사용.
- **부모 식별**: `parent_page_id IS NULL` + `deleted_at IS NULL` + `title === label` + `icon === emoji` 매칭. 여러 개면 `created_at asc` 첫 번째.
  - 사용자가 부모 title을 바꾸면 다음 클릭 시 새 부모가 생기는 동작이 직관적.
- **자식 title 규칙**:
  - 메모: `YYYY-MM-DD HH:mm`
  - 일기: `YYYY-MM-DD` (하루 1건)
  - 기도제목: `YYYY-MM-DD HH:mm`
- **자식 icon**: 부모와 동일 emoji 상속.
- **일기 중복 처리**: 오늘 날짜 자식이 이미 있으면 그 페이지로 이동 (재생성 X). 메모/기도제목은 분 단위라 거의 충돌 없음 — 충돌 시 그냥 또 생성.

### 13.2 구현
새 Server Action: `lib/actions/quick-create.ts`
```ts
export async function createQuickPage(
  template: 'memo' | 'diary' | 'prayer'
): Promise<ActionResult<{ pageId: string }>>
```

내부 흐름:
1. `getSessionUser()` 인증 가드
2. `findRootByTitleIcon(supabase, label, icon)` — `pages.select(id).is(parent_page_id, null).is(deleted_at, null).eq(title).eq(icon).order(created_at).limit(1)`
3. 부모 없으면 INSERT 후 id 확보
4. 일기인 경우: `pages.select(id).eq(parent_page_id, parent).eq(title, today).is(deleted_at, null).limit(1)` — 있으면 그 id 반환
5. 자식 INSERT (`parent_page_id`, `title`, `icon`, `position` = nextPosition)
6. `revalidatePath('/', 'layout')` + return `{ pageId }`

### 13.3 QuickCreate 변경
- import `createPage` 제거 → `createQuickPage` import
- handler 단순화: `const r = await createQuickPage(t); if (r.ok) router.push('/p/' + r.data.pageId);`
- 일기 title 계산은 Server Action 내부로 이동 (클라이언트 today를 굳이 보낼 필요 없음 — UTC 차이는 사용자 경험상 무시)

### 13.4 테스트
- `lib/__tests__/actions-quick-create.test.ts`: Zod template 검증, 부모 신규 생성 vs 재사용, 일기 중복 시 기존 반환 (mock supabase chain)
- `QuickCreate.dom.test.tsx`: `createPage` mock → `createQuickPage` mock 교체. assert 호출 인자 단순화 (template string만).
