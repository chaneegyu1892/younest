# M2.3 검색 (Cmd+K 모달 + `/search` 결과 페이지) — 설계 스펙

> 작성일: 2026-05-13
> 단계: M2.3 (M2 에디터 핵심의 sub-phase)
> 의존: M2.1 페이지 CRUD, M2.2 BlockNote 에디터(`pages.content jsonb`)
> 관련 PRD: §5.9 검색, §12.3 검색 모달, S-014 검색 결과

---

## 1. 목적

`pages.title` + `pages.content`(BlockNote jsonb)에서 키워드 검색을 제공한다.

- 빠른 탐색: `Cmd/Ctrl+K` 전역 모달 → 즉시 미리보기 → 클릭 시 페이지 이동
- 상세 탐색: `/search?q=...` 결과 페이지 → 정렬 토글, 페이지네이션
- 본문 검색은 평탄화 컬럼 + GIN 트라이그램 인덱스로 빠르게
- 비공개 페이지(M5)와의 정합: 평탄화 컬럼이 NULL이면 자연스럽게 서버 검색에서 제외

## 2. 스코프

### 포함

- Cmd+K 전역 검색 모달
- `/search?q=&sort=&page=` 결과 페이지 (S-014)
- 제목 + 본문(content_text) ILIKE 검색
- 정렬: 관련도순(제목 매치 우선) / 최근 수정순
- 페이지네이션: 모달 limit 10, 결과 페이지 limit 30
- 휴지통(`deleted_at IS NOT NULL`) 자동 제외
- 사이드바 상단 검색 버튼 (페이지 트리 위, ⌘K 힌트 포함)
- 본문 평탄화 트리거 + GIN 인덱스

### 제외 (의도적)

- **본문 매치 스니펫·하이라이트** — B 스코프 결정. 결과 행은 제목 + breadcrumb만
- **즐겨찾기/타입 필터** — 1인 사용 페이지 규모상 과함
- **비공개 페이지 클라이언트 검색** — M5에서 본격 (이번엔 평탄화 컬럼 NULL 처리만)
- **백링크/멘션 검색** — M4 데이터베이스 시점에 재검토
- **검색어 자동완성·이력** — v2
- **PostgreSQL `tsvector` + 한국어 토크나이저** — `pg_trgm`으로 충분. 페이지 수 수천 넘어가면 도입 검토

## 3. 시스템 구조


## 4. 데이터 모델

### 마이그레이션 `20260514xxxxxx_pages_content_text_search.sql`

```sql
create extension if not exists pg_trgm;

alter table pages add column content_text text;

create or replace function extract_blocknote_text(doc jsonb)
returns text
language plpgsql
immutable
as $$
declare
  result text;
begin
  begin
    select string_agg(t::text, ' ')
    into result
    from jsonb_path_query(doc, 'strict $.**.text ? (@.type() == "string")') as t;
    return result;
  exception when others then
    return null;
  end;
end
$$;

create or replace function pages_content_text_trigger()
returns trigger
language plpgsql
as $$
begin
  -- M5: is_private = true면 NULL 유지 (서버 검색 자동 제외)
  if new.content is null then
    new.content_text := null;
  else
    new.content_text := extract_blocknote_text(new.content);
  end if;
  return new;
end
$$;

create trigger pages_content_text_sync
  before insert or update of content on pages
  for each row execute function pages_content_text_trigger();

-- 백필
update pages set content_text = extract_blocknote_text(content) where content is not null;

create index pages_title_trgm_idx on pages using gin (title gin_trgm_ops);
create index pages_content_text_trgm_idx on pages using gin (content_text gin_trgm_ops);
```

**RPC `search_pages`** — 관련도순 정렬을 PostgREST에서 표현하기 까다로워 SQL에서 처리:

```sql
create or replace function search_pages(
  q text,
  sort_mode text default 'relevance',  -- 'relevance' | 'recent'
  result_limit int default 30,
  result_offset int default 0
)
returns setof pages
language sql
stable
as $$
  select *
  from pages
  where deleted_at is null
    and owner_id = auth.uid()
    and (
      title ilike '%' || q || '%'
      or content_text ilike '%' || q || '%'
    )
  order by
    case when sort_mode = 'relevance' then
      case when title ilike '%' || q || '%' then 0 else 1 end
    end,
    updated_at desc
  limit result_limit
  offset result_offset
$$;
```

`security definer` 사용 안 함 — `auth.uid()`는 호출자 컨텍스트, RLS는 SECURITY INVOKER 기본값으로 작동.

### 비공개 페이지(M5) 정합

- M5 도입 시 `is_private = true`인 행은 트리거에서 `content_text := null`로 분기 추가
- ILIKE는 NULL 매칭 안 되므로 자동 제외
- PRD §8.7 "비공개 페이지는 서버 측 전문 검색이 불가능" 자연 충족

## 6. UX 디테일

### Cmd+K 모달

- 진입: `Cmd/Ctrl+K` (글로벌 keydown), 사이드바 SearchTrigger 클릭
- 빈 상태: 최근 수정 페이지 5개 표시 (`order by updated_at desc limit 5`)
- 입력: 200ms debounce → `searchPagesAction(q, { limit: 10 })`
- 키보드:
  - ↑↓: 행 이동
  - Enter: 선택 행의 페이지로 이동 (`router.push('/p/${id}')`)
  - ⌘Enter: `/search?q=${q}`로 이동
  - Esc: 모달 닫기
- 결과 행: 아이콘(emoji 또는 기본 📄) + 제목 + 부모 경로 breadcrumb (작게)
- 결과 0개: "결과 없음" 단순 텍스트
- 모달 하단: "전체 결과 보기 →" 링크 (입력값 있을 때만)
- 모달 너비: 600px, 위쪽에서 20vh 위치 (Cmd-K 표준)

### `/search?q=&sort=&page=` (S-014)

- 헤더: `"<q>" 검색 결과 N개` (N은 첫 페이지에선 정확, 페이지네이션에선 "추가 결과 있음" 정도)
- 정렬 토글: "관련도순" / "최근 수정순" (URL 파라미터로 상태)
- 결과 리스트: SearchResultRow 재사용
- 페이지네이션: limit 30, 하단 "이전 / 다음" — offset 기반
- 빈 결과: "검색 결과가 없습니다"
- 빈 쿼리(`?q=` 또는 미지정): `/dashboard`로 리다이렉트

### 사이드바 검색 버튼

- 위치: 사이드바 상단 (페이지 트리 위, 사용자 메뉴 아래)
- 디자인: 아이콘 + "검색" 텍스트 + 오른쪽 작은 `⌘K` 힌트
- 클릭: `useSearchModal().open()`

## 7. API 계약

### `lib/search/search-pages.ts`

```ts
export type SearchSort = 'relevance' | 'recent';

export interface SearchOpts {
  limit?: number;       // default 30
  offset?: number;      // default 0
  sort?: SearchSort;    // default 'relevance'
}

export interface SearchHit {
  id: string;
  title: string;
  icon_emoji: string | null;
  parent_id: string | null;
  updated_at: string;
  // breadcrumb은 클라이언트에서 트리 컨텍스트로 계산 (sidebar tree와 동일)
}

export async function searchPages(
  q: string,
  opts: SearchOpts = {}
): Promise<SearchHit[]>;
```

내부:
- 빈 `q` → `[]` 반환 (DB 호출 안 함)
- `q` 200자 초과 → 200자로 cut
- `escapeIlike(q)`로 `%` `_` `\` escape
- supabase `rpc('search_pages', { q, sort_mode, result_limit, result_offset })`

### `lib/search/actions-search.ts`

```ts
'use server';
export async function searchPagesAction(input: {
  q: string;
  limit?: number;
  sort?: SearchSort;
}): Promise<{ hits: SearchHit[] } | { error: string }>;
```

Zod 스키마로 검증, 인증 실패 시 `error: 'unauthorized'`.

## 8. 에러 처리 / 엣지 케이스

| 케이스 | 처리 |
|---|---|
| 빈 검색어 | `[]` 반환 (DB 호출 없음). `/search?q=` → `/dashboard` 리다이렉트 |
| 200자 초과 | Zod에서 cut 또는 reject (cut 채택) |
| ILIKE 메타문자 | `q.replace(/[\\%_]/g, '\\$&')` |
| 트리거 함수 예외 | `exception when others then return null` — `content_text` NULL, 저장은 성공 |
| 마이그레이션 백필 실패 | 백필은 마이그레이션 안에 포함, 실패 시 마이그레이션 자체가 롤백 |
| RLS 우회 시도 | `lib/search/`는 anon/sessioned client만 사용 (service-role 금지). 정적 테스트로 보장 |
| 동시 검색 요청 | debounce + 이전 요청 결과 무시 (AbortController) |
| 트리 컨텍스트 없는 검색 결과 행 | breadcrumb은 옵셔널 — parent_id만 있으면 "..." 또는 부모 1단계만 |

## 9. 테스트 계획

### 단위 (vitest)

### 컴포넌트 RTL

- `SearchResultRow`: 행 렌더링 + 클릭 콜백 (1)

## 10. 마이그레이션 / 실행 순서

1. **SQL 마이그레이션 작성** (`20260514xxxxxx_pages_content_text_search.sql`)
2. **사용자가 Supabase MCP로 적용** — content_text 컬럼, 트리거, 인덱스, RPC, 백필
3. **타입 재생성** — `content_text` 컬럼 반영
4. **백엔드 레이어** — `lib/search/` 작성 + 단위 테스트
5. **검색 모달 + 사이드바 통합** — `SearchModalProvider`, `SearchModal`, `SearchTrigger`
6. **결과 페이지** — `/search` RSC + 정렬·페이지네이션
7. **수동 검증** — 제목/본문 매치, 정렬, ⌘K, ⌘Enter, breadcrumb, 휴지통 제외

## 11. 보안 / RLS

- 모든 검색 쿼리는 `pages` RLS 정책 통과 → `auth.uid()` 격리 자동
- `search_pages` RPC는 `security invoker` (기본값) — RLS 자연 적용
- service-role 키 사용 금지 — 정적 테스트로 강제
- 검색어 로깅 절대 X (사용자 콘텐츠 일부일 수 있음)
- SQL injection: PostgREST/RPC는 파라미터화 자동, ILIKE 메타문자만 별도 escape

## 12. 향후 (M2.3 이후)

- **M2.4** 휴지통 UI, drag&drop, 동적 RLS 시나리오 테스트
- **M5** 비공개 페이지 → 클라이언트 측 검색 폴백
- **확장 시점 도입 후보**: tsvector + 한국어 토크나이저, 검색 이력, 자동완성, 백링크 검색

## 결정 로그

| 결정 | 대안 | 채택 이유 |
|---|---|---|
| 평탄화 컬럼 + GIN trgm | 매번 jsonb_path_query / 클라이언트 측 | 트리거 비용 1회, 인덱스 가속, M5 NULL 처리 자연스러움 |
| RPC `search_pages` | PostgREST `.or()` | 관련도 정렬을 SQL `CASE WHEN`으로 표현. 호출부 단순 |
| 모달 limit 10 / 페이지 limit 30 | 무한 스크롤 | 1인 사용 규모상 페이지네이션이 단순하고 충분 |
| ⌘Enter → 결과 페이지 | 모달에서 무한 스크롤 | 결과 페이지 분리는 PRD 결정 (S-014). 모달은 빠른 진입 전용 |
| 스니펫·하이라이트 X | 표시 | B 스코프 결정. 본문 텍스트 추출과 매치 위치 계산 비용 vs UX 이득 비대칭 |
| `pg_trgm` ILIKE | `tsvector` + 한국어 토크나이저 | 1인 MVP 데이터 규모, 한국어 토크나이저(`pg_jieba`/`mecab-ko`) 설치/유지 비용 |
