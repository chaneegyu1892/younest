# M2.2 — BlockNote 본문 에디터 + 자동저장 (Design)

**Date:** 2026-05-13
**Phase:** M2.2 (M2 sub-phase, M2.1 페이지 CRUD + 사이드바 트리 완료 후속)
**Status:** Design approved, ready for implementation plan

---

## 1. 목표

`app/(app)/p/[pageId]/PageBodyPlaceholder.tsx`를 실제 BlockNote 에디터로 교체하고, 본문 변경 시 디바운스 500ms로 자동저장한다.
- 기본 블록 + 커스텀 블록 3종(toggle / callout / pageLink) 동작
- 헤더에 저장 상태(저장 중 / 저장됨 · n초 전 / 저장 실패) 표시
- 페이지 이탈·재시도·실패 복구 처리
- M5(E2E 암호화) 도입 시 호출부 변경 0이 되도록 추상화 계층 미리 배치

비범위:
- 이미지 블록 (M3)
- 전문 검색 / Cmd+K (M2.3 또는 M3)
- 휴지통 UI / 블록 드래그 핸들 외 페이지 트리 drag & drop (M2.4)
- 비공개 페이지 / E2E 암호화 (M5)
- 멀티 디바이스 실시간 동기화 (M6 Supabase Realtime)

## 2. 결정 사항 요약

| 항목 | 결정 | 근거 |
|---|---|---|
| 저장 단위 | `pages.content jsonb` + `pages.content_encrypted bytea` 컬럼 추가. `blocks` 테이블은 M2.2~M4 미사용 | BlockNote가 트리 전체를 한 단위로 관리. 페이지 단위 PUT이 단순하고 비공개(M5) `editor.document` 통째 암호화와 자연스럽게 정합. PRD §8.3과 일관 |
| 자동저장 | 500ms 디바운스, 낙관적, 실패 시 1s→2s→4s 백오프 3회 재시도, 영구 토스트로 수동 재시도 | PRD §5.4. 일시 네트워크 장애 자동 복구 |
| Flush 트리거 | 라우트 변경 + `beforeunload` + 컴포넌트 unmount 시 즉시 호출 | 데이터 손실 방지 |
| dirty 표시 | 페이지 헤더 우측 `SaveStatusBadge` (독립 컴포넌트). "저장 중…" → "저장됨 · n초 전" → 1분 후 "저장됨" 고정 | 노션·옵시디언 패턴. PageHeader는 M2.1 안정 상태 유지 |
| 신규 페이지 초기 콘텐츠 | DB는 `content = null`. 첫 onChange 전에는 저장 안 함. 에디터는 빈 paragraph 1개로 시작 | 빈 페이지는 굳이 빈 JSON으로 채우지 않음 |
| 한글 IME | 추가 처리 없음 (M0 PoC `roundtrip.test.ts`에서 회귀 방지) | M0에서 검증 완료 |
| 커스텀 블록 | M0 `lib/blocknote/schema.ts` 그대로 사용. 슬래시 메뉴에 3종 노출 | PoC 결과물 재사용 |
| `pageLink` 선택 UI | M2.2에 포함. `PageLinkPickerModal` 신규 작성 (M2.1 `MovePageModal` 패턴 재사용). 자기 자신/조상 제외 규칙 **없음** (자기 참조 허용) | 노션과 동일. 선택 UI 없으면 dead UI |
| 충돌 처리 | last-write-wins. 멀티 디바이스 동기화는 M6 | M2.1과 동일 정책. 단일 디바이스 가정 |
| 어드민 격리 | Server Action에서 service-role 미사용. RLS 의존. 정적 검증 1건 추가 | M2.1 패턴 |

## 4. 컴포넌트 구조

```
app/(app)/p/[pageId]/
├── page.tsx                       (RSC, fetchPage → content 전달)
├── PageHeader.tsx                 (기존, 변경 최소)
├── PageBody.tsx                   (NEW, 'use client', dynamic import wrapper)
└── BlockNoteEditor.tsx            (NEW, 'use client', 실제 에디터 + 슬래시 메뉴)

components/editor/
├── SaveStatusBadge.tsx            (NEW, status 표시)
├── PageLinkPickerModal.tsx        (NEW, MovePageModal 패턴 재사용)
└── slash-menu.ts                  (NEW, 커스텀 블록 3종을 BlockNote 슬래시 메뉴에 추가)

lib/blocknote/
├── schema.ts                      (기존, 변경 없음)
├── custom-blocks/                 (기존 toggle/callout/page-link)
├── useAutosave.ts                 (NEW, 디바운스+재시도+flush hook)
└── content-codec.ts               (NEW, JSON ↔ DB 컬럼 추상화. M5 분기 지점)

lib/supabase/actions/
└── pages-content.ts               (NEW, updatePageContent)

supabase/migrations/
└── 20260513140000_pages_content.sql  (NEW)
```

### 4.1 책임 분담

- **`page.tsx` (RSC)**: `fetchPage(pageId)` → `<PageHeader>` + `<PageBody initialContent={page.content} pageId={page.id}>`. DB 접근은 여기서만.
- **`PageBody.tsx`**: `next/dynamic`으로 `BlockNoteEditor`를 `ssr: false`로 로드. 로딩 중 skeleton.
- **`BlockNoteEditor.tsx`**: BlockNote 인스턴스 생성 + schema 적용 + 슬래시 메뉴 주입. `onChange` → `useAutosave`. `pageLink` 블록 클릭 시 `PageLinkPickerModal` 오픈. status를 헤더로 전달 (Context 또는 외부 store 미도입 — pageId 기반의 작은 in-module event emitter로 처리, 상세는 §5).
- **`useAutosave`**: `(pageId, content) → { status, flush, retry }`. 핵심 로직 집중.
- **`SaveStatusBadge`**: `useAutosave`의 status를 구독해 헤더에 렌더. PageHeader는 M2.1 안정 상태를 유지, badge는 별도 슬롯에 옆에 둔다.
- **`content-codec.ts`**: M2.2에서는 `serialize(doc) = doc`, `deserialize(jsonb) = jsonb`. **M5에서 여기에 `encryptIfPrivate(doc, dek)` 분기 추가** — 호출부는 변경 0.

### 4.2 dynamic import 이유

BlockNote / Mantine은 `window` 의존. SSR 시점에 import만 해도 깨짐. `next/dynamic({ ssr: false, loading: Skeleton })`로 격리.

### 4.3 SaveStatusBadge ↔ Editor 연결

`useAutosave`는 `BlockNoteEditor` 내부에서 호출되므로 status는 Editor 안에 있다. `SaveStatusBadge`는 PageHeader 옆 슬롯(`page.tsx`)에 마운트되어 트리 위치가 다르다. 두 가지 옵션:

1. **(채택) `BlockNoteEditor`가 `SaveStatusBadge`를 자식으로 렌더**: PageBody가 `<BlockNoteEditor>` 마운트 시 portal 또는 단순한 컴포넌트 합성으로 헤더에 badge를 끼워 넣지 않고, `BlockNoteEditor` 컴포넌트 트리 내부 상단에 badge를 렌더. 위치는 우상단 sticky. PageHeader에는 변경 없음.

페이지 헤더와 badge가 시각적으로 가까이 있되, 트리 구조상 분리되어 PageHeader는 M2.1 그대로 유지된다.

## 5. 데이터 흐름

### 5.1 페이지 로드

```
URL /p/[pageId]
  → page.tsx (RSC)
  → fetchPage(pageId) — pages 1행 (content jsonb 포함)
  → <PageHeader/> + <PageBody initialContent={page.content} pageId={page.id}/>
  → PageBody → dynamic import BlockNoteEditor (ssr:false)
  → BlockNoteEditor 마운트:
       initialContent === null   → 빈 paragraph 1개로 시작
       initialContent !== null   → deserializeContent(initialContent) → editor.replaceBlocks()
  → hasLoadedSuccessfully = true  (저장 차단 해제)
```

`hasLoadedSuccessfully` 플래그: 로드 실패 또는 deserialize 실패 시 첫 onChange가 잘못된 데이터로 DB를 덮어쓰는 것을 방지.

### 5.2 편집 (낙관적 저장)

```
사용자 입력
  → BlockNote 내부 상태 갱신 (즉시 화면 반영)
  → editor.onChange 발화
  → useAutosave: latestDoc 캡처, 500ms 디바운스 타이머 reset
  → 디바운스 만료:
       status: idle → saving
       serializeContent(doc) → updatePageContent(pageId, content)
       성공: status saved + lastSavedAt = now
       실패: 백오프 재시도 (1s, 2s, 4s)
       3회 모두 실패: status error, 영구 토스트 "저장 실패 — 다시 시도" 버튼
  → 재시도 도중 새 입력이 들어오면:
       latest로 덮어쓰고 디바운스 다시 시작 (이전 실패 폐기)
```

### 5.3 Flush 트리거 (디바운스 무시 즉시 저장)

- **라우트 변경**: `app/(app)/p/[pageId]/PageBody.tsx`의 `useEffect` cleanup에서 `flush()`. pathname 변경 시 cleanup이 실행됨을 활용.
- **`window.beforeunload`**: `useAutosave` 내부에서 리스너 등록. pending이 있으면 `fetch(..., { keepalive: true })`로 동기 비스무리하게 호출. best-effort — 브라우저가 이행을 보장하지 않음.
- **컴포넌트 unmount**: cleanup에서 `flush()`.

### 5.4 status 전이

```
idle
  → (입력) → saving (디바운스 만료 시점)
  → (성공) → saved → (1s tick) → "저장됨 · n초 전" → (60s) → "저장됨" 고정
  → (실패 3회) → error → (사용자 클릭) → saving …
```

`saved`/`error` 모두에서 새 입력이 들어오면 바로 `idle`로 돌아가 디바운스 시작.

### 5.5 `pageLink` 블록 흐름

```
슬래시 메뉴 "/page" 선택
  → 빈 pageLink 블록 삽입 (props.pageId: null)
  → 블록 자체가 "페이지 선택" 버튼 UI 렌더
  → 클릭 → PageLinkPickerModal 오픈
       (현재 사용자의 모든 페이지 검색, 제외 규칙 없음)
  → 선택 시 editor.updateBlock(blockId, { props: { pageId: selectedId } })
  → 블록이 선택된 페이지 제목·아이콘으로 렌더 (클라이언트 사이드 fetch)
  → 블록 클릭 시 router.push(`/p/${pageId}`)
```

- 페이지 제목·아이콘 fetch는 단순한 client-side `fetchPage`. SWR/TanStack 미도입(M2.1 정책 동일).
- 깨진 참조(삭제됨, RLS 차단): "삭제된 페이지" 회색 표시, 클릭 무효.

## 6. 에러 처리

| 위치 | 시나리오 | 처리 |
|---|---|---|
| 저장 | 네트워크 일시 장애 | 1s → 2s → 4s 백오프 3회. 시각적으로는 `saving` 유지(깜빡임 방지) |
| 저장 | 3회 모두 실패 | status `error`, 영구 토스트 "저장 실패 — 다시 시도" 버튼. 클릭 시 즉시 재시도 |
| 저장 | 재시도 중 새 입력 | latest로 덮어쓰고 디바운스 다시 시작. 이전 실패 폐기 |
| 저장 | 인증 만료 (401) | 토스트 "다시 로그인하세요" + 재시도 중단 |
| 로드 | `fetchPage` 실패 | 기존 `notFound()` 흐름 유지 |
| 로드 | `content`가 null (신규) | 빈 paragraph 1개로 시작. 정상 |
| 로드 | `content`가 잘못된 형식 | try/catch로 빈 paragraph 폴백 + 콘솔 경고. `hasLoadedSuccessfully = false`로 첫 onChange까지 저장 차단 → 데이터 보존 |
| pageLink | 참조 페이지 삭제됨 / RLS 차단 | "삭제된 페이지" 회색 표시, 클릭 무효 |
| Server Action | Zod 검증 실패 | 400 + 메시지 |
| Server Action | 미인증 | 401 |
| Server Action | RLS 차단 (소유자 아님) | 404 (행 미발견과 구분 안 함, 정보 누출 방지) |

**로깅 규칙**: 사용자 콘텐츠는 로그에 절대 남기지 않음. `pageId`와 에러 코드만. PRD 보안 원칙.

## 7. 테스트 전략

### 7.1 단위 (vitest)

- **`useAutosave`** (가장 중요):
  - 디바운스 500ms 정확성 (`vi.useFakeTimers()`)
  - 연속 입력 시 마지막 값만 저장 (호출 1회)
  - 성공 → status `saved` 전이
  - 실패 시 3회 백오프 (1s, 2s, 4s) 확인
  - 3회 실패 후 status `error` + 수동 재시도 동작
  - `flush()` 즉시 호출 (라우트 변경 시뮬레이션)
  - unmount 시 pending flush
- **`content-codec.ts`**: `serialize(deserialize(x)) === x` 라운드트립 (M5 분기 가능성 회귀 방지)
- **`updatePageContent` Server Action**: Zod 검증 케이스 (uuid/null/array, 실패 케이스) — M2.1의 `actions-pages.test.ts` 패턴 그대로

### 7.2 컴포넌트 RTL (happy-dom)

- `SaveStatusBadge`: status별 텍스트 렌더
- `PageLinkPickerModal`: 검색 입력 → 결과 필터링, 선택 → onChoose 콜백
- `BlockNoteEditor` 자체는 1회 mount 테스트만 (initialContent 반영 확인). 깊은 인터랙션 테스트 안 함 — BlockNote 라이브러리 책임

### 7.3 RLS 정적 검증 확장

- `actions/pages-content.ts`가 service-role을 사용하지 않는지 확인 1건 추가 (M2.1 패턴)

### 7.4 제외

- E2E (Playwright) — M2 전체 종료 시 (M2.4) 또는 별도 단계
- 동적 RLS (사용자 A vs B 격리) — M5에서 supabase test client 도입 시
- 한글 IME — M0 `roundtrip.test.ts`가 회귀 방지

### 7.5 합격 기준

- 모든 단위/RTL 테스트 PASS
- `npm run typecheck`, `npm run lint` clean
- `npm run build` 성공
- 수동 검증:
  - 빈 페이지 생성 → 첫 입력 → 1초 내 "저장됨" 표시
  - 새로고침 후 본문 유지
  - 네트워크 차단 → "저장 실패" 토스트 → 복구 후 재시도 시 정상
  - 페이지 이동 시 pending 변경이 사라지지 않음
  - `pageLink` 블록 삽입 → 페이지 선택 → 클릭으로 이동
  - 한글 IME 빠른 입력 시 글자 유실 없음

## 8. M5(E2E 암호화) 도입 동선

M2.2에서 미리 배치하는 추상화:

- `content-codec.ts` 인터페이스:
  ```ts
  serializeContent(doc, opts?: { dek?: CryptoKey }) → { plain?: jsonb; encrypted?: bytea }
  deserializeContent(row, opts?: { dek?: CryptoKey }) → doc
  ```
  M2.2에서는 `dek` 미사용, `plain` 분기만. M5에서 `dek` 전달 시 `encrypted` 분기 추가.
- `updatePageContent` Server Action은 `{ content?, content_encrypted? }`를 모두 받도록 시그니처 설계. M2.2에서는 `content_encrypted: null`로 호출, M5에서는 분기.
- 페이지 비공개 토글(M5)은 `pages.is_private` 플래그 + 컬럼 swap 트랜잭션.

## 9. 의존성 / 위험

- **BlockNote 0.50.x**: M0에서 PoC 통과. Mantine 8.x 페어링 유지 ([[project-next15-bundled-react-useeffectevent]]).
- **happy-dom 한계**: BlockNote 깊은 인터랙션 테스트는 jsdom조차 불안정. RTL은 mount/snapshot 수준까지만.
- **`beforeunload` 신뢰성**: 브라우저 정책상 보장 없음. 라우트 변경 cleanup이 1차 방어선, `beforeunload`는 보조.
- **큰 페이지 페이로드**: 페이지 단위 PUT이므로 큰 페이지일수록 페이로드 증가. 현실적으론 1인 노트 수십KB 이내 — 미래 1MB 초과 시 chunking 도입 가능.
