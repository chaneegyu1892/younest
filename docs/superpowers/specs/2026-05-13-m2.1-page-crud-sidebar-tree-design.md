## 1. 아키텍처 개요

### 렌더링 모델
- `app/(app)/layout.tsx` (서버 컴포넌트) → Supabase 쿼리로 사용자 페이지 트리를 **한 번** 페치 → flat 배열을 `<Sidebar pages={...} />` / `<SidebarMobile pages={...} />`에 전달.
- `Sidebar` / `SidebarMobile`은 **client component** (트리 nest, 펼침 상태 localStorage, 컨텍스트 메뉴, 낙관적 업데이트 필요).
- `app/(app)/p/[pageId]/page.tsx` (서버) → 해당 페이지 row 단일 페치 → `<PageHeader>` (client) + `<PageBodyPlaceholder>` 표시.
- 모든 mutation은 **Server Action**으로 `lib/actions/pages.ts`에 모음. `useOptimistic` + `useTransition`으로 낙관적 업데이트, 실패 시 토스트 + 자연 롤백.

### 데이터 모델 활용
기존 `pages` 테이블 그대로 사용. **마이그레이션 추가 없음.**

사용 컬럼:
- `id, user_id, parent_page_id, type='document', title, icon, is_favorite, position, deleted_at, created_at, updated_at`

M2.1에서는 다음 컬럼을 손대지 않음:
- `title_encrypted` (M5)
- `cover_url` (M3)
- `is_private` (M5 — M2.1에서는 항상 `false`로 생성)

### 상태 분류

| 상태 | 위치 | 이유 |
|---|---|---|
| 페이지 트리 데이터 | DB → RSC → Sidebar props | 진실의 원천 |
| 펼침/접힘 | localStorage (`younest:tree-expanded`) | 디바이스별 UI 상태 |
| 컨텍스트 메뉴 열림/포커스 | React state | ephemeral |
| 낙관적 트리 변경 | `useOptimistic` | mutation in-flight |
| 토스트(삭제 Undo 등) | `sonner` (shadcn `<Toaster>`) | 일시적 알림 |

### 의존성 추가
- `emoji-mart` + `@emoji-mart/react` + `@emoji-mart/data` — 이모지 피커 (≈100KB gzip, 한국어 i18n 지원).
- `sonner` — 토스트 (미설치 시 추가).
- **추가 없음**: dnd-kit (M2.4), TanStack Query (불필요), React Query (불필요).

### 라우팅 변경 없음
기존 `(app)` 그룹 · `/p/[pageId]` 동적 라우트 그대로 사용.

## 3. 데이터 플로우 & Server Action 시그니처

### Server Actions (`lib/actions/pages.ts`)
모두 `'use server'`, 입력은 Zod 검증, 인증은 Supabase 세션 기반(서버 헬퍼). 반환은 `Promise<ActionResult<T>>` 형식:

```ts
type ActionResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: string };
```

**시그니처**:

```ts
// 입력 스키마 예시
const createSchema = z.object({
  parentPageId: z.string().uuid().nullable(),
  type: z.enum(['document']),  // M2.1은 document만
  title: z.string().max(200).optional(),
});

// 액션들 (모두 Promise<ActionResult<T>>)
createPage(input)        → { id, parent_page_id, position, type, title, icon, is_favorite }
renamePage(id, title)    → { id, title }
setPageIcon(id, emoji)   → { id, icon }  // emoji가 null이면 제거
toggleFavorite(id)       → { id, is_favorite }
movePage(id, newParentId, newPosition?) → { id, parent_page_id, position }
softDeletePage(id)       → { deletedIds: string[] }   // 자기 + 모든 후손
restorePage(deletedIds)  → { restoredIds: string[] }  // Undo용
```

### 사이클·권한·position 규칙
- `movePage`: 새 parent가 자기 자신이거나 자기 후손이면 reject (`error: 'cycle'`). 후손 탐색은 `lib/pages/tree.ts`의 순수 함수로 클라/서버 양쪽 검증.
- 모든 쿼리는 `user_id = auth.uid()` 자동(RLS) — Server Action에서 추가 verify는 안 함(RLS 신뢰).
- `position` 계산: 새 페이지는 같은 parent에서 `max(position) + 1` (빈 경우 0). 이동 시 `newPosition` 미지정이면 새 parent의 끝으로. drag&drop 없으니 단순 정수로 충분(M2.4에서 fractional indexing으로 마이그레이션 시점에 재검토).

### Soft delete cascade
`softDeletePage(id)`는 RPC 또는 단일 트랜잭션으로:
1. CTE로 후손 ID 전부 수집 (같은 사용자, `deleted_at IS NULL`).
2. 한 번에 `UPDATE pages SET deleted_at = now() WHERE id IN (...)`.

반환된 `deletedIds`는 Undo에서 그대로 사용.

### 낙관적 업데이트 패턴 (Sidebar에서)
```ts
const [optimisticTree, applyOptimistic] = useOptimistic(
  serverTree,
  (current, mutation: OptimisticMutation) => reduceTree(current, mutation),
);

async function handleRename(id: string, title: string) {
  startTransition(async () => {
    applyOptimistic({ kind: 'rename', id, title });
    const res = await renamePage(id, title);
    if (!res.ok) toast.error('이름 변경 실패 — 다시 시도해주세요');
    // 성공 시 RSC 재페치가 권위 있는 상태로 자연 수렴
  });
}
```

- 모든 mutation은 동일 패턴.
- 트리 reducer는 `lib/pages/tree.ts`의 순수 함수 (테스트 용이).

### Revalidation 전략
- 모든 mutation 후 Server Action 안에서 `revalidatePath('/(app)', 'layout')` 호출 → 사이드바·페이지 헤더 모두 재페치.
- `/(app)/p/[pageId]`도 `revalidatePath('/(app)/p/[pageId]', 'page')` 추가 호출 (rename, setIcon, toggleFavorite만 필요).

### 삭제 + Undo 흐름
1. **자식 없음** → 즉시 `softDeletePage` → 토스트 `"휴지통으로 이동했어요"` + Undo 버튼(5초).
2. **자식 있음** → `<DeleteConfirmModal>` 열어 `"하위 페이지 N개도 같이 이동돼요"` 확인 → `softDeletePage` → 토스트 + Undo.
3. **Undo 클릭** → `restorePage(deletedIds)` → reverse cascade.

### 빈 상태 데이터 흐름
- 트리 fetch 결과 `pages.length === 0` → `<Sidebar>`는 `<EmptyPagesState>`, 대시보드는 `<EmptyDashboard>` 렌더.
- 두 빈 상태의 "+ 새 페이지" 버튼 모두 `createPage({parentPageId: null, type: 'document'})` 호출 → 성공 시 `router.push('/p/${id}')` + 제목 자동 포커스.

## 4. 에러 처리 & 엣지케이스

### Server Action 에러 처리
모든 액션은 `{ ok, error?, data? }` 형식. 클라이언트는 `!res.ok` 시 토스트 + 낙관적 변경 자동 롤백 (`useOptimistic`은 transition 종료 시 서버 상태로 자연 수렴).

### 주요 엣지케이스 매트릭스

| 케이스 | 검증 위치 | 처리 |
|---|---|---|
| 이름 빈 문자열 | 클라이언트 + Zod | 허용하되 표시는 `"제목 없음"`으로 fallback (사이드바·페이지 헤더 모두) |
| 이름 200자 초과 | Zod | reject, 토스트 `"제목은 200자 이내로 입력해주세요"` |
| `movePage`: 자기 자신/후손을 parent로 | 클라/서버 양쪽 (`isDescendant()`) | 클라에서 모달 옵션 비활성화. 서버에서도 reject (`error: 'cycle'`). |
| `movePage`: 이미 삭제된 페이지 | RLS + `deleted_at IS NULL` 필터 | reject |
| 동시 편집 충돌 (두 디바이스에서 같은 페이지 이름 변경) | 마지막 쓰기 우선 | 의도적 — 1인 사용자 한정. 실시간 동기화는 M6 이후. |
| 페이지 삭제 후 그 페이지가 현재 라우트인 경우 | client | 토스트 + `router.push('/dashboard')` |
| Undo 토스트 만료 후 클릭 | sonner | 토스트 dismiss로 자연 처리 |
| Undo 중 새로 삭제됨 | 무관 | restore는 `deletedIds`만 복구 |
| 후손 cascade 삭제 중 일부 row가 이미 삭제됨 (레이스) | 멱등 | `WHERE deleted_at IS NULL` 가드 + 반환된 IDs만 Undo |
| 이모지 피커: 이모지가 아닌 입력 | 클라이언트 | emoji-mart가 emoji만 반환. `setPageIcon(id, null)`로 제거 가능. |
| 트리 fetch에 고아 노드 (`parent_page_id`가 가리키는 부모가 삭제됨) | `lib/pages/tree.ts` | 고아 발견 시 루트로 승격하여 표시 + 콘솔 warn (production은 빈 로그) — 사용자 데이터 잃지 않음 우선 |
| localStorage 비활성화/오류 | `expanded-state.ts` | try/catch로 SSR-safe + 실패 시 in-memory fallback. 모든 노드 펼친 상태로 시작. |
| RLS 위반 응답 (타 사용자 페이지 접근 시도) | RLS | 0 row 반환 → 클라이언트는 `"페이지를 찾을 수 없어요"` 표시 |
| 페이지 0개 상태에서 첫 페이지 생성 | empty state → 트리 | revalidate로 자연 전환 |

### 보안·RLS 가드
- 기존 RLS 정책(M1.4에서 검증된 22건)에 변경 없음. 새 테이블·새 컬럼 없음.
- Server Action에서 `service-role` client 사용 **금지**. 일반 server client만 사용 (`auth.uid()` 컨텍스트). M1.4 정적 테스트 위반 안 함.
- `lib/actions/pages.ts`가 `pages` 테이블에 `from()` 호출하는 것은 정상 (어드민 콘솔 코드 아님). M1.4 grep 룰은 어드민 디렉토리 한정 — 그대로 통과.

### IME / 한글 입력
인라인 제목 편집은 `<input>` 사용 (textarea 아님). `compositionstart` / `compositionend`로 IME 입력 중 Enter 무시. PoC에서 검증된 패턴 재사용.

### 모바일 컨텍스트 메뉴
우클릭이 없는 환경 → **길게 누르기(long-press, 500ms)** 로 컨텍스트 메뉴 오픈. `⋯` 아이콘은 hover/터치 모두 노출.

### 의도적으로 처리하지 않는 케이스 (범위 밖)
- 비공개 페이지 (`is_private=true`) 생성/암호화 → M5
- 페이지 본문(blocks) → M2.2
- 드래그 앤 드롭 재배치 → M2.4
- 휴지통 UI (복구/영구삭제) → M2.4. 사이드바 "🗑️ 휴지통" 링크는 현재 placeholder 유지.
- 검색 (⌘K) → M3 이후
- 다중 디바이스 실시간 동기화 → M6 이후

### 수동 검증 체크리스트 (M2.1 완료 조건)
빌드/타입/린트/테스트 GREEN + 다음 시나리오 본인 1회 확인:

- [ ] 빈 상태에서 "+ 새 페이지" → 페이지 생성 → `/p/[id]`로 라우팅 + 제목 포커스
- [ ] 사이드바 페이지 더블클릭 → 인라인 편집 → 변경 즉시 본문 헤더에도 반영
- [ ] 본문 헤더에서 이름변경 → 사이드바도 즉시 반영 (낙관적)
- [ ] hover "+" → 자식 페이지 생성 → 부모 자동 펼침
- [ ] 우클릭 → 컨텍스트 메뉴 6개 항목 모두 노출
- [ ] 이모지 피커 → 선택 → 사이드바·본문 헤더 둘 다 이모지 표시 / 제거 가능
- [ ] 즐겨찾기 토글 → 즐겨찾기 섹션에 추가 + 원래 위치에도 유지 (중복 노출)
- [ ] 자식 없는 페이지 삭제 → 즉시 사라짐 + 토스트 Undo → 복구
- [ ] 자식 있는 페이지 삭제 → 모달 "N개 같이 이동" → 확인 → 부모+자식 모두 사라짐 → Undo로 트리 통째 복구
- [ ] 이동 모달: 자기 자신/후손 선택 불가, "최상위로 이동" 동작
- [ ] 새로고침 후 펼침/접힘 상태 유지 (localStorage)
- [ ] 모바일(햄버거) → 동일 트리/모든 액션 동작
- [ ] 한글 IME로 제목 편집 시 변환 중 Enter가 저장 안 함
- [ ] 두 디바이스에서 동시 이름변경 → 마지막 쓰기가 이김 (예상된 동작)

### E2E 테스트 (Playwright)
M2.1에서는 **추가하지 않음**. 사유:
- 페이지 트리 핵심 로직(reducer, tree util)이 순수 함수로 분리되어 단위 테스트로 충분히 커버됨
