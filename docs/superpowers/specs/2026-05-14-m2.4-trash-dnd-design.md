## 1. 목적

- **사이드바 페이지 트리 DnD** — 노션 수준 UX (같은 부모 reorder + 다른 페이지로 drop = 자식 + 루트 영역 drop = 루트)
- **휴지통 UI** — `/trash` (S-015)에서 삭제된 페이지 목록 + 복원 + 영구삭제 + 일괄 작업 + 검색
- M2.1에서 "M2.4로 보류"라고 표시했던 두 부채를 청산
- 동적 RLS 시나리오 테스트는 본 phase 범위 밖 (E2E 도입 시 = M5)

## 2. 스코프

### 포함

- `pages.position` 컬럼 `int → double precision` 마이그레이션
- Fractional positioning util (`positionBetween`)
- `dnd-kit` 기반 사이드바 DnD (`@dnd-kit/core` + `@dnd-kit/sortable`)
- 3-zone drop target (above/body/below) 모델
- `isDescendant()` 사이클 차단 (UI disable + 서버 reject)
- `MovePageModal` 유지 (DnD fallback)
- `movePage` Server Action 시그니처: `newPosition: number.finite()` (정수 제약 제거)
- `/trash` 페이지 (S-015) — flat 리스트, 제목 ILIKE 검색, 삭제일 desc 정렬
- 일괄 선택 체크박스 + 일괄 복원/영구삭제
- 영구삭제 확인 모달 (`DeleteConfirmModal` M2.1 패턴 재사용)
- `hardDeletePages` Server Action + `hard_delete_pages` RPC
- 새 단위/RTL 테스트 (~25건 추가)

### 제외 (의도적)

- **Fractional position rebalance 자동화** — 1인 사용 규모상 100번쯤 같은 사이에 끼우지 않으면 발생 불가. 발생 시 `nextPosition` fallback + `console.warn`만. M3 이후 도입 검토.
- **동적 RLS 시나리오 테스트** (사용자 A vs B 격리) — Supabase 두 세션 인증 인프라 필요. M5(E2E) 시점.
- **휴지통 자동 정리 (N일 후 자동 영구삭제)** — cron 정책. M5 이후.
- **"전체 비우기" 버튼** — 위험도 크고 실수 비용 큼. 일괄 선택으로 충분.
- **들여쓰기까지 정밀 드래그** (노션 풀 동작) — 자식 만들기는 body zone에 drop으로 단순화. 노션 80% UX.
- **모바일 DnD** — 모바일은 MovePageModal 그대로. dnd-kit의 PointerSensor가 touch도 처리하지만 좁은 사이드바에 정밀 drag UX 만들기 비쌈.

### 마이그레이션 2: 영구삭제 RPC

`supabase/migrations/20260514xxxxxx_pages_hard_delete.sql`:

```sql
create or replace function hard_delete_pages(p_ids uuid[])
returns int
language plpgsql
security invoker
as $$
declare
  affected int;
begin
  delete from pages
  where id = any(p_ids)
    and user_id = auth.uid()
    and deleted_at is not null;
  get diagnostics affected = row_count;
  return affected;
end
$$;

grant execute on function hard_delete_pages(uuid[]) to authenticated;
```

**가드**: `user_id = auth.uid()` (RLS 보강) + `deleted_at is not null` (살아있는 페이지 보호).

## 5. Fractional position 알고리즘

### `lib/pages/fractional-position.ts`

```ts
const REBALANCE_THRESHOLD = 1e-9;
const FRACTIONAL_FALLBACK = 1;

export function positionBetween(
  before: number | null,
  after: number | null,
): number {
  if (before === null && after === null) return FRACTIONAL_FALLBACK;
  if (before === null) return after! - 1;
  if (after === null) return before + 1;

  const result = (before + after) / 2;
  if (Math.abs(after - before) < REBALANCE_THRESHOLD || !Number.isFinite(result)) {
    // 정밀도 한계 — 호출자가 nextPosition fallback 사용해야 함을 알리려고 throw
    throw new FractionalPrecisionError(before, after);
  }
  return result;
}

export class FractionalPrecisionError extends Error {
  constructor(public before: number, public after: number) {
    super(`positionBetween precision limit: before=${before}, after=${after}`);
    this.name = "FractionalPrecisionError";
  }
}
```

### Rebalance 전략 (현 단계)

발생 빈도 0에 가까움. 발생 시:
1. `FractionalPrecisionError` catch
2. `console.warn`으로 한 번 로그 (메타데이터만, 사용자 콘텐츠 X)
3. `nextPosition(siblings, parentId)` (max + 1) fallback — 트리 맨 끝으로 이동
4. 사용자가 의도한 위치는 아니지만 동작 보장

자동 rebalance(같은 부모의 모든 형제 재정렬) 는 별도 phase.

## 6. DnD 알고리즘

### Drop zone 3종

```
┌─────────────────────────┐
│ above zone (4px)        │  ← drop = 같은 부모, 현재 노드 바로 위
├─────────────────────────┤
│ body                    │  ← drop = 현재 노드의 자식 (마지막)
│  [icon] Title           │
├─────────────────────────┤
│ below zone (4px)        │  ← drop = 같은 부모, 현재 노드 바로 아래
└─────────────────────────┘
```

루트 영역 (`RootDropZone`): 트리 컨테이너 하단 빈 영역. drop = 루트 트리의 마지막 자식.

### `lib/pages/dnd-calculate.ts`

```ts
export interface DropResult {
  newParentId: string | null;
  newPosition: number;
}

export function calculateDrop(args: {
  draggedId: string;
  targetId: string | null;        // null = RootDropZone
  zone: "above" | "below" | "body";
  pages: readonly PageNode[];
}): DropResult | { error: "cycle" | "self" };
```

**알고리즘**:
1. `targetId === draggedId` → error: "self"
2. `zone === "body"`이고 `isDescendant(pages, draggedId, targetId)` → error: "cycle"
3. zone에 따라 `newParentId` 결정:
   - `above`/`below` → target의 parent_page_id
   - `body` → target.id (null이면 루트, 즉 RootDropZone)
4. 새 부모의 형제 정렬해서 before/after 페이지 찾음
5. `positionBetween(before, after)` 호출, 예외 시 `nextPosition` fallback

### 사이클 차단 시각화

`onDragOver`에서 `isDescendant()` 체크 → 후손 노드들에 `cursor: not-allowed` + drop zone 숨김. dnd-kit `Modifier` 또는 `useDroppable.disabled`로 처리.

## 7. movePage Server Action 수정

```ts
// lib/actions/pages.ts
const moveSchema = z.object({
  id: z.string().uuid(),
  newParentId: z.string().uuid().nullable(),
  newPosition: z.number().finite(),  // ← .int().nonnegative() 제거
});
```

`newPosition` optional 제거 — DnD에선 항상 계산해서 보냄. MovePageModal은 `nextPosition()` 호출 결과 전달 (변경 없음).

서버 측 사이클 차단: 기존 `isDescendant()` 호출 그대로 유지. Zod 통과 후 사이클이면 reject.

`movePage` 호출부:
- `MovePageModal` — 시그니처 동일, position 계산 방식만 nextPosition으로 유지
- `PageTreeClient.onDragEnd` — `calculateDrop()` 결과 사용

## 8. 휴지통 페이지 (S-015)

### `app/(app)/trash/page.tsx`

```tsx
interface PageProps {
  searchParams: Promise<{ q?: string }>;
}

export default async function TrashPage({ searchParams }: PageProps) {
  const { q } = await searchParams;
  const trimmed = (q ?? "").trim();
  const pages = await fetchDeletedPages(trimmed || undefined);
  return <TrashListClient pages={pages} initialQ={trimmed} />;
}
```

### `lib/supabase/queries/pages.ts` 확장

```ts
export async function fetchDeletedPages(q?: string): Promise<PageNode[]> {
  const supabase = await createSupabaseServerClient();
  let query = supabase
    .from("pages")
    .select(PAGE_SELECT)
    .not("deleted_at", "is", null)
    .order("deleted_at", { ascending: false })
    .limit(200);

  if (q) {
    const escaped = escapeIlike(q.slice(0, 200));  // M2.3 escape util 재사용
    query = query.ilike("title", `%${escaped}%`);
  }

  const { data, error } = await query;
  if (error) throw error;
  return (data ?? []) as unknown as PageNode[];
}
```

`PAGE_SELECT`에 `deleted_at` 추가 필요. 현재 빠짐.

### `TrashListClient` UI

- 상단: `TrashSearchInput` (debounce 200ms, URL 파라미터 push)
- 선택 N≥1: "선택 N개 / [복원] [영구삭제]" 일괄 작업 바
- 리스트: `TrashRow * N`
  - 체크박스 (Set\<string\> 상태)
  - 아이콘 + 제목 (no link, 휴지통은 진입 불가)
  - 삭제일 (date-fns 또는 단순 toLocaleDateString)
  - 개별 [복원] [영구삭제] 버튼
- 빈 결과: "휴지통이 비어있습니다" / 검색 결과 없으면 "검색 결과가 없습니다"

### `hardDeletePages` Server Action

```ts
const hardDeleteSchema = z.array(z.string().uuid()).min(1);

export async function hardDeletePages(
  ids: string[],
): Promise<ActionResult<{ deletedCount: number }>> {
  const user = await getSessionUser();
  if (!user) return { ok: false, error: "unauthorized" };

  const parsed = hardDeleteSchema.safeParse(ids);
  if (!parsed.success) return { ok: false, error: "invalid_input" };

  const supabase = await createSupabaseServerClient();
  const { data, error } = await supabase.rpc("hard_delete_pages", {
    p_ids: parsed.data,
  });
  if (error) return { ok: false, error: "delete_failed" };

  revalidatePath("/trash");
  return { ok: true, data: { deletedCount: data ?? 0 } };
}
```

### 복원

`restorePage(ids)` 그대로 사용. 복원 후:
- `revalidatePath("/trash")` (서버 액션 내부)
- `revalidatePath("/p")` — 사이드바 트리 갱신
- 토스트 "N개 항목 복원됨"

### 영구삭제 확인 모달

`DeleteConfirmModal` (M2.1) 재사용. props 확장:
```tsx
<DeleteConfirmModal

  message="이 페이지를 영구히 삭제합니다. 복구할 수 없습니다."
  confirmLabel="영구 삭제"
  destructive  // 빨강 강조
  onConfirm={() => hardDeletePages(selectedIds)}

```

18. 전체 검증 (typecheck/lint/test/build) + 메모리 업데이트

## 13. 보안 / RLS

- 모든 쿼리는 sessioned client (RLS 자동)
- `hard_delete_pages` RPC: `security invoker` + `user_id = auth.uid()` + `deleted_at is not null` 이중 가드
- 영구삭제는 service-role 절대 사용 X — 정적 테스트(`lib/__tests__/rls-coverage.test.ts`)가 `lib/actions/pages.ts` 검증으로 자동 커버
- DnD는 read-only 트리 + Server Action으로 mutation. 클라이언트 측 position 계산은 서버에서 Zod로 검증 후 RLS 통과해야 적용
- 동적 RLS 시나리오 (사용자 A가 B 페이지 영구삭제 시도) 는 M5 E2E에서 추가

## 14. 향후 (M2.4 이후)

- **M3** 대시보드 + 이미지 자동 압축
- **M4** 데이터베이스 시스템
- **M5** E2E 암호화 + 동적 RLS 시나리오 테스트 (Supabase test client 또는 별도 인증 helper)
- **Fractional rebalance 자동화**: 같은 부모에서 100번 이상 끼우기 발생 시점 (실제 사용 데이터로 임계치 결정)
- **휴지통 자동 정리**: 30일 후 자동 영구삭제 cron (Supabase pg_cron 또는 Vercel cron)
